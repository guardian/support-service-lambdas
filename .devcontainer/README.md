# Dev container config

This folder contains the configuration to work in a [dev container](https://containers.dev/);
an isolated, reproducible development environment that runs inside a container, typically Docker.

The configuration skeleton is generated by the Guardian's [devenv](https://github.com/guardian/devenv) config tool.

## Getting started (macOS)

These steps take you from a fresh checkout to a running dev container.

### Prerequisites

- **[Docker Desktop](https://www.docker.com/products/docker-desktop/)** — must be installed and running. The dev
  container runs inside Docker.
- An IDE with dev container support — either **[IntelliJ IDEA](https://www.jetbrains.com/idea/)** (via JetBrains
  Gateway / the built-in dev containers support) or **[VS Code](https://code.visualstudio.com/)** with the
  **[Dev Containers extension](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-containers)**.
  These are assumed to be already installed.

All other tooling (Java, Node, `mise`, the GitHub Copilot CLI, etc.) is installed automatically inside the container,
so nothing else needs to be installed on your host.

### Open the dev container

1. Clone the repo and open it in your IDE:
   ```bash
   git clone git@github.com:guardian/support-service-lambdas.git
   cd support-service-lambdas
   ```
2. Start the dev container from your IDE:
   - **VS Code**: open the repo folder, then run **Dev Containers: Reopen in Container** from the command palette
     (or click **Reopen in Container** when prompted).
   - **IntelliJ IDEA**: open `.devcontainer/shared/devcontainer.json`, then use the gutter action / dev container
     prompt to **Create Dev Container and Mount Sources**.
3. Wait for the first build to finish. The first run is slower because it provisions the shared Scala caches
   (Ivy and coursier volumes), sets up the in-container tooling, and installs the GitHub Copilot CLI. Subsequent
   starts reuse these and are much faster.

### Working inside the container

Once the container is running, open a terminal inside it and install the Node dependencies:

```bash
pnpm install
```

From here, follow the main [README](../README.md) for the Node.js and Scala development workflows (building, testing,
linting, etc.).

### Opening a shell

To open an interactive shell inside the running dev container from your host, use the helper script (the container
must already be started from your IDE):

```bash
./.devcontainer/devsh.sh
```

### Troubleshooting

If the container fails to build or set up correctly, the setup output is logged inside the container at
`/var/log/post-create.log`, which is a good starting point for debugging.

## .devcontainer directory contents

### `README.md`

Edit this file as you see fit.  It is created on the first run of `devenv init` but it won't be overwritten
by subsequent runs unless you delete it entirely.

### `devenv.yaml`

The source-of-truth configuration for this project's dev container. Edit this file to add/remove devenv modules,
or any other config settings. Then run: 
```bash
devenv generate
``` 
to regenerate the shared `devcontainer.json` file.

### `shared/devcontainer.json`

The main devcontainer configuration, committed to the repository and shared by all developers.
It is generated from `devenv.yaml`.

> [!IMPORTANT]
> Do not edit this file directly! Instead, edit `devenv.yaml` and then run `devenv generate`.

### `user/devcontainer.json`

A personal copy of the devcontainer config, **git-ignored** (see `.gitignore`).
This file is intended for your own overrides:
extra VS Code extensions or Intellij plugins, personal settings or tweaks that shouldn't affect other contributors.
Modify this file freely without worrying about polluting shared config.
> [!IMPORTANT]
> But be aware that `devenv generate` will overwrite it!

### `devsh.sh`

A helper script to open an interactive shell inside this project's running dev container.
It finds the running container by matching the current git repo bind mount, detects the
container user, and execs bash in the mounted project directory. Works for both JetBrains
and VSCode dev containers.

Usage (the dev container must already be running - start it from your IDE first):
```bash
./.devcontainer/devsh.sh
```
